# DSL2React 项目指导总纲

**项目：** DSL2React v2.0
**日期：** 2026-04-29
**基于文档：** 头脑风暴记录、架构设计、产品需求文档

---

## 一、项目定位

DSL2React 是一个将 MasterGo 设计稿 DSL 自动转换为高质量 React 代码的工具。核心解决的问题是：**长设计稿通过 LLM 直接生成 HTML/React 时质量差**。

解决思路用一句话概括：
> 先结构化，再分块；先抽 token，再生成；先局部生成，再全局组装；先截图对比，再局部修正。

---

## 二、核心架构：五层管道

```
┌──────────────────────────────────────────────────────┐
│  Layer 1 — 解析层 (Parser)                            │
│  DSL 输入 → AST 构建 → 语法验证                        │
├──────────────────────────────────────────────────────┤
│  Layer 2 — 分析层 (Analyzer)                          │
│  Design Tokens 提取 → Section 智能识别 → 依赖图构建     │
├──────────────────────────────────────────────────────┤
│  Layer 3 — 生成层 (Generator)                         │
│  逐 Section 调用 LLM → 并行生成 → HTML/JSX 片段        │
├──────────────────────────────────────────────────────┤
│  Layer 4 — 组装层 (Assembler)                         │
│  按拓扑排序拼接 → 应用 Design Tokens → 完整页面         │
├──────────────────────────────────────────────────────┤
│  Layer 5 — 验证层 (Validator)                         │
│  截图对比 → 差异检测 → 自动修正（最多 3 轮）             │
└──────────────────────────────────────────────────────┘
```

**设计原则：**
- 纯内存存储，零数据库依赖，一条命令启动
- 会话驱动，数据仅在会话期间有效，支持随时重置
- 增量优先，只重新生成变更部分

## 三、核心数据流

### 完整生成流程

```
用户上传 MasterGo DSL
  → [解析层] 构建 AST
  → [分析层] 提取 Design Tokens + 识别 Sections + 生成 Manifest
  → [生成层] 并行调用 LLM 生成各 Section 的 HTML（带 data-dsl-id 属性）
  → [用户] 通过浏览器插件可视化编辑 HTML，保存 Patch
  → [组装层] 应用 Patch → 拼接 Final HTML → 生成 React 组件
  → [验证层] Puppeteer 截图 → Pixelmatch 对比 → 差异 >5% 则 LLM 局部修正
  → 输出最终 React 代码
```

### 增量更新流程

```
用户应用 Patch
  → 识别变更的 Section
  → 查询依赖图（BFS），找出所有受影响的 Sections
  → 只重新生成受影响的 Sections
  → 重新组装 → 输出
```

## 四、七大关键模块

| # | 模块 | 职责 | 关键技术点 |
|---|------|------|-----------|
| 1 | **DSL Parser** | 词法/语法分析，构建 AST | AST 方式替代正则，支持错误定位 |
| 2 | **Token Extractor** | 提取颜色/字体/间距等，生成 CSS 变量 | 相似值聚类（如 #333 ≈ #323232），CSS Variables |
| 3 | **Section Analyzer** | 智能识别 Section，计算复杂度，构建依赖图 | 布局层级 + 间距分析 + 语义标签，拓扑排序 |
| 4 | **LLM Generator** | 构建 prompt，调用 LLM，并行生成 | 按复杂度调参，few-shot 示例，重试机制 |
| 5 | **Code Assembler** | 拼接 Section，应用 Tokens，输出 React | CSS 作用域隔离，统一命名规范 |
| 6 | **Visual Validator** | 截图对比，差异热力图 | Puppeteer + Pixelmatch，智能容差 |
| 7 | **Correction Engine** | 基于 diff 局部修正，收敛检测 | 最多 3 次，差异值不下降则停止 |

## 五、关键设计决策

### 5.1 为什么分 Section？
长设计稿直接丢给 LLM，token 数过大、上下文混乱，生成质量急剧下降。分块后每个 Section 独立生成，复杂度可控，还能并行加速。

### 5.2 为什么先抽 Design Tokens？
各 Section 独立生成时容易出现样式不一致（同一颜色写成不同值）。提前提取 Tokens 生成 CSS 变量，所有 Section 引用变量而非硬编码，保证全局一致性。

### 5.3 为什么用 HTML 作为中间产物？
HTML 是用户可直接在浏览器中编辑的格式。通过 `data-dsl-id` 属性保持与原始 DSL 的映射关系，用户编辑后的 Patch 可以精确回溯到对应节点。

### 5.4 为什么纯内存不用数据库？
单用户本地工具，会话驱动，不需要持久化历史。纯内存方案零配置、零 I/O 延迟、一条命令启动。进程重启数据丢失是可接受的。

### 5.5 为什么截图对比而非 DOM 对比？
最终用户关心的是视觉效果，不是 DOM 结构。截图对比直接衡量"看起来像不像"，是最接近用户感知的验证方式。

## 九、核心风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| Section 识别不准确 | 高 — 组件被拆散或混杂 | 多维度识别（层级+间距+语义），提供人工调整入口 |
| LLM API 不稳定/超时 | 高 — 生成流程中断 | 重试机制（3 次）、超时控制、备选 API |
| Design Token 提取不完整 | 中 — 样式不一致 | 全节点遍历 + 相似值聚类 + 人工补充 |
| 修正陷入循环 | 中 — 流程无法结束 | 最多 3 次修正，差异值不下降则停止 |
| 内存溢出（大型设计稿） | 中 — 进程崩溃 | 流式处理、限制最大 Section 数、内存监控 |
| CSS 命名冲突 | 低 — 样式覆盖 | CSS Modules / scoped styles / BEM 命名 |

## 十一、开发指引

1. **每个模块对应一个清晰的接口（Interface）**，模块间通过接口通信，便于独立测试和替换。
2. **先写接口，再写实现**。所有核心模块的 TypeScript Interface 已在架构文档中定义。
3. **LLM 调用不做缓存**，每次重新生成。简化架构，避免缓存一致性问题。
4. **测试优先**。每个模块完成后立即编写单元测试，目标覆盖率 >80%。
5. **增量开发**。按 Phase 顺序推进，每个 Phase 结束时应有可运行的中间产物。
6. **Prompt 工程是核心竞争力**。LLM 生成质量直接取决于 prompt 设计，需要持续迭代优化。
