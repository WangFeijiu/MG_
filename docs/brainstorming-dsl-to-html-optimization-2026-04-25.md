# 头脑风暴会话：优化 DSL 到 HTML 的生成流程

**日期：** 2026-04-25
**项目：** DSL2React
**目标：** 优化 MasterGo DSL 到 HTML 的生成流程，解决长设计稿生成质量差的问题

## 会话背景

### 当前问题
- DSL 直接通过 LLM 生成 HTML，设计稿很长时效果差
- 缺乏结构化的生成策略
- 没有质量验证和修正机制

### 期望方案
采用"先结构化，再分块；先抽 token，再生成；先局部生成，再全局组装；先截图对比，再局部修正"的工程方法

### 目标流程
```
MasterGo DSL
  ↓
DSL 预处理
  ↓
提取全局 Design Tokens
  ↓
识别页面 Section
  ↓
生成 section-manifest.json
  ↓
逐 section 生成 HTML
  ↓
统一 CSS / 组件规范
  ↓
拼接页面
  ↓
截图对比
  ↓
让 LLM 基于差异修正局部 section
```

## 关键洞察

### 洞��� 1：分层生成架构是核心
**描述：** 采用"解析 → 结构化 → 分块生成 → 组装 → 验证"的五层架构，每层职责清晰

**来源：** SCAMPER（修改生成顺序）、Starbursting（流程分解）、反向分析（依赖图构建）

**影响：** 高 | **工作量：** 高

**为什么重要：** 这是整个优化方案的基础架构，决定了系统的可维护性和扩展性。清晰的分层可以让每个模块独立优化和测试。

**实施建议：**
- 定义清晰的接口和数据格式
- 每层独立测试和优化
- 支持插件化扩展

---

### 洞察 2：智能 Section 识别是关键瓶颈
**描述：** Section 识别的准确性直接影响生成质量，需要结合语义分析、视觉分组和可选的人工标注

**来源：** 反向分析（识别错误风险）、Starbursting（谁决定边界）、SCAMPER（调整识别策略）

**影响：** 高 | **工作量：** 中

**为什么重要：** Section 划分不当会导致组件被拆散或不相关内容混在一起，严重影响生成质量。这是"先结构化，再分块"的核心步骤。

**实施建议：**
- 基于布局层级（容器嵌套深度）
- 基于间距（大间距作为分界）
- 基于语义标签（header/section/article）
- 提供可视化工具让设计师调整边界

---

### 洞察 3：Design Tokens 是全局一致性保障
**描述：** 完整提取并统一应用 Design Tokens（颜色、字体、间距等），使用 CSS 变量实现

**来源：** SCAMPER（消除重复）、Starbursting（包含什么）、反向分析（提取不完整风险）

**影响：** 高 | **工作量：** 中

**为什么重要：** 避免每个 Section 独立生成样式导致的不一致问题，这是"先抽 token，再生成"的核心价值。

**实施建议：**
- 遍历所有节点提取样式值
- 聚类相似值（如 #333 和 #323232）
- 生成 CSS 变量定义
- 在生成时引用变量而非硬编码

---

### 洞察 4：LLM 调用需要精细化控制
**描述：** 根据 Section 复杂度动态调整参数、使用 few-shot、批量并行处理、结果缓存

**来源：** SCAMPER（调整参数）、Starbursting（如何优化）、反向分析（超时风险）

**影响：** 高 | **工作量：** 中

**为什么重要：** LLM 调用是成本和时间的主要消耗点，优化调用策略可以显著提升性能和降低成本。

**实施建议：**
- 简单 Section（如纯文本）降低 temperature
- 复杂 Section（如表单）提高 max_tokens
- 提供 2-3 个高质量示例
- 无依赖的 Section 并行生成
- 基于 DSL 哈希缓存结果

---

### 洞察 5：截图对比需要智能容差
**描述：** 使用结构对比 + 像素对比的混合方式，根据 Section 类型动态调整容差阈值

**来源：** SCAMPER（组合对比方式）、反向分析（误报风险）、Starbursting（输出什么）

**影响：** 中 | **工作量：** 中

**为什么重要：** 过于严格会导致误报，过于宽松会漏掉真实问题。智能容差是"先截图对比，再局部修正"的前提。

**实施建议：**
- 文本内容：严格对比（低容差）
- 图片/图标：宽松对比（高容差）
- 布局结构：中等容差
- 输出差异热力图辅助人工判断

---

### 洞察 6：修正机制需要收敛保证
**描述：** 限制修正次数、记录修正历史、检测循环修正，避免无限循环

**来源：** 反向分析（修正循环风险）、Starbursting（何时触发修正）

**影响：** 中 | **工作量：** 低

**为什么重要：** 防止系统陷入反复修正的死循环，保证流程能够正常结束。

**实施建议：**
- 最多修正 3 次
- 记录每次修正的差异值
- 如果差异值不再下降，停止修正
- 提供人工介入选项

---

### 洞察 7：依赖图管理支持增量更新
**描述：** 构建 Section 依赖图，按拓扑排序生成，支持局部修改时的增量更新

**来源：** 反向分析（依赖混乱）、SCAMPER（增量生成）、Starbursting（manifest 作用）

**影响：** 中 | **工作量：** 中

**为什么重要：** 支持设计稿局部修改时只重新生成受影响的 Section，大幅提升迭代效率。

**实施建议：**
- 在 manifest 中记录 Section 依赖关系
- 使用拓扑排序确定生成顺序
- DSL 变更时，只重新生成受影响的 Section
- 缓存未变更的 Section 结果
