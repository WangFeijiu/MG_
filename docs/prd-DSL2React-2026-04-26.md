# DSL2React 产品需求文档 (PRD)

**项目：** DSL2React
**版本：** 2.0
**日期：** 2026-04-26
**产品经理：** Product Manager
**项目级别：** Level 2（中型项目，5-15 个故事）

---

## 1. 执行摘要

DSL2React 是一个将 MasterGo 设计稿 DSL 转换为高质量 React 代码的自动化工具。系统采用**五层管道架构**，核心流程为：DSL → 可编辑 HTML → 插件编辑 → Final HTML → React 代码。通过分块处理、设计令牌提取、LLM 并行生成和自动化截图对比修正，解决了长设计稿生成质量差的问题。

**核心价值：**
- 设计稿到代码的自动化转换，大幅减少手动切图和编码工作
- 可视化 HTML 编辑中间产物，让用户在生成 React 代码前精确调整
- 增量更新机制，只重新生成变更部分，节省时间和成本
- 自动化质量验证，确保生成代码与设计稿的视觉一致性

## 2. 商业目标

### BG-001: 自动化设计到代码转换
**目标：** 将设计稿快速准确地转换为可用的 React 代码，减少 80% 的手动编码时间。

### BG-002: 解决长设计稿生成质量差
**目标：** 通过智能 Section 分块和并行处理，支持大型设计稿的高质量转换。

### BG-003: 提升设计还原的视觉准确度
**目标：** 通过截图对比和自动修正，将设计还原度提升到 95% 以上。

### 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 设计还原度 | >95% | 截图对比差异 <5% |
| 生成速度（单 Section） | <10s | 计时统计 |
| 生成速度（完整页面） | <2min | 计时统计 |
| 增量更新效率 | <全量的30% | 时间对比 |
| 测试覆盖率 | >80% | Vitest 覆盖率报告 |

## 3. 用户画像

### 主要用户：前端开发者
- **角色：** 需要将设计稿转换为 React 代码的开发者
- **需求：** 快速、准确地将 MasterGo 设计稿转为可用的 React 组件
- **痛点：** 手动还原设计稿耗时长、容易出错、长页面尤其困难

### 次要用户：UI/UX 设计师
- **角色：** 使用 MasterGo 的设计师
- **需求：** 验证设计稿是否被正确还原，快速调整细节
- **痛点：** 与开发者沟通设计还原问题的成本高

## 4. 核心用户流程

```
用户上传 MasterGo DSL
    ↓
系统解析 DSL → 构建 AST
    ↓
提取 Design Tokens + 识别 Sections
    ↓
生成可编辑 HTML（带 data-dsl-id 属性）
    ↓
用户通过浏览器插件可视化编辑 HTML
  - 调整圆角、位置、尺寸、裁剪等
  - 保存 Patch
    ↓
系统应用 Patch → 生成 Final HTML
    ↓
基于 Final HTML 生成 React 组件代码
    ↓
截图对比验证 → 自动修正（如需要）
    ↓
输出最终 React 代码
```

**增量更新流程：**
```
用户应用补丁（Patch）
    ↓
识别变更的 Section
    ↓
查询依赖图，找出受影响的 Sections
    ↓
只重新生成受影响的 Sections
    ↓
重新组装 → 输出更新后的代码
```

## 5. 功能需求

### FR-001: DSL 解析
**优先级：** Must Have

**描述：** 系统能解析 MasterGo DSL 并构建抽象语法树（AST）。

**验收标准：**
- [ ] 实现 DSL 词法分析器
- [ ] 实现 DSL 语法分析器
- [ ] 构建 AST 数据结构
- [ ] 实现 AST 遍历接口
- [ ] 支持错误处理和位置信息

---

### FR-002: Design Tokens 提取
**优先级：** Must Have

**描述：** 系统能从 DSL 中提取设计令牌（颜色、字体、间距等），并生成 CSS 变量定义。

**验收标准：**
- [ ] 遍历 AST 提取颜色、字体、间距等样式值
- [ ] 实现相似值聚类算法
- [ ] 生成 CSS 变量定义
- [ ] 构建 Token 索引

---

### FR-003: Section 智能识别
**优先级：** Must Have

**描述：** 系统能智能识别页面 Section，将长页面拆分为可管理的生成单元。

**验收标准：**
- [ ] 基于布局层级识别 Section
- [ ] 基于间距识别 Section 边界
- [ ] 基于语义标签识别 Section
- [ ] 计算 Section 复杂度评分
- [ ] 生成 section manifest

---

### FR-004: 依赖图管理
**优先级：** Must Have

**描述：** 系统能构建 Section 依赖图，支持增量更新的变更传播识别。

**验收标准：**
- [ ] 构建 Section 依赖图数据结构
- [ ] 实现拓扑排序算法
- [ ] 实现受影响 Section 识别算法（BFS）

---

### FR-005: DSL → 可编辑 HTML 生成
**优先级：** Must Have

**描述：** 系统能将 DSL 直接转换为带 `data-dsl-id` 属性的可编辑 HTML，作为中间产物供浏览器插件编辑。

**验收标准：**
- [ ] 生成完整的 HTML 页面（含布局、样式、图片）
- [ ] 每个节点带有 data-dsl-id、data-dsl-type、data-dsl-name 属性
- [ ] Flex 容器正确渲染（gap、direction、align、justify）
- [ ] 图片节点正确渲染（img 标签 + object-fit）
- [ ] 文字节点正确渲染（字体、颜色、行高、对齐）
- [ ] 圆角、padding、margin、阴影等样式正确

---

### FR-006: 浏览器插件编辑
**优先级：** Must Have

**描述：** 系统提供 Chrome 浏览器插件，支持在生成的 HTML 中可视化编辑节点属性。

**验收标准：**
- [ ] 点击 HTML 元素选中并编辑
- [ ] 支持圆角调整（联动/独立四角）
- [ ] 支持位置调整（X/Y 坐标）
- [ ] 支持尺寸调整（宽度、高度、gap）
- [ ] 支持裁剪设置（overflow、object-fit）
- [ ] 保存 Patch 并导出 Patch JSON

---

### FR-007: Final HTML → React 代码生成
**优先级：** Must Have

**描述：** 系统能基于编辑后的 Final HTML 生成 React 组件代码。

**验收标准：**
- [ ] 解析 Final HTML 结构和样式
- [ ] 生成可用的 React 组件（JSX/TSX）
- [ ] 应用 Design Tokens（CSS 变量）
- [ ] 统一 CSS 命名规范
- [ ] 生成完整的可导入组件

---

### FR-008: 截图对比验证
**优先级：** Must Have

**描述：** 系统能通过截图对比验证生成代码与设计稿的视觉一致性。

**验收标准：**
- [ ] 集成 Puppeteer 渲染代码
- [ ] 集成 Pixelmatch 对比截图
- [ ] 生成差异热力图
- [ ] 计算差异百分比
- [ ] 识别问题区域坐标

---

### FR-009: 自动修正引擎
**优先级：** Must Have

**描述：** 系统能基于差异分析自动调用 LLM 修正代码，最多 3 次迭代。

**验收标准：**
- [ ] 分析差异区域
- [ ] 构建修正 prompt
- [ ] 调用 LLM 局部修正
- [ ] 实现修正收敛检测（差异值持续下降）
- [ ] 记录修正历史

---

### FR-010: 增量更新
**优先级：** Must Have

**描述：** 系统能通过补丁方式增量更新 DSL，只重新生成变更部分。

**验收标准：**
- [ ] 实现补丁解析
- [ ] 识别变更的 Section
- [ ] 查询依赖图找出受影响的 Sections
- [ ] 只重新生成受影响的 Sections
- [ ] 重新组装代码

---

### FR-011: 核心 API 端点
**优先级：** Must Have

**描述：** 系统提供 RESTful API 端点支持完整工作流。

**验收标准：**
- [ ] POST /api/v1/dsl - 上传 DSL
- [ ] POST /api/v1/generate - 创建生成任务
- [ ] GET /api/v1/generate/:jobId - 查询状态
- [ ] GET /api/v1/generate/:jobId/result - 获取结果
- [ ] POST /api/v1/patch - 应用补丁
- [ ] POST /api/v1/reset - 重置会话
- [ ] WebSocket 实时进度推送

---

### FR-012: 智能容差算法
**优先级：** Should Have

**描述：** 系统能根据 Section 类型动态调整截图对比容差。

- [ ] 容差配置可调

## 7. Epics

### EPIC-001: DSL 解析与分析

**描述：** 实现从 MasterGo DSL 到结构化数据的完整解析管道，包括 AST 构建、Design Tokens 提取和 Section 智能识别。

**功能需求：** FR-001, FR-002, FR-003, FR-004

**Story 估算：** 5-8 stories

**优先级：** Must Have

**商业价值：** 这是整个系统的基础，所有后续功能都依赖于此 Epic 的完成。

---

### EPIC-002: HTML 生成与编辑

**描述：** 实现 DSL 到可编辑 HTML 的转换，以及浏览器插件的交互式编辑能力。

**功能需求：** FR-005, FR-006

**Story 估算：** 3-5 stories

**优先级：** Must Have

**商业价值：** HTML 作为中间可编辑产物，是连接设计稿和最终代码的关键桥梁。用户可以在生成 React 代码前精确调整样式。

**描述：** 实现从 Final HTML 到 React 组件代码的转换，以及基于依赖图的增量更新机制。

**功能需求：** FR-007, FR-010

**Story 估算：** 4-6 stories

**优先级：** Must Have

**商业价值：** React 代码生成是核心输出，增量更新大幅提升迭代效率。

---

### EPIC-004: 质量验证与修正

**描述：** 实现截图对比验证、智能容差和自动修正引擎，确保生成代码的视觉准确性。

**功能需求：** FR-008, FR-009, FR-012

**Story 估算：** 4-6 stories

**优先级：** Must Have

**商业价值：** 质量保证是用户信任的基础，自动修正减少人工干预。

---

### EPIC-005: API 与集成界面

**功能需求：** FR-011, FR-013

**Story 估算：** 3-5 stories

**优先级：** Should Have

**商业价值：** API 支持工作流集成，Web 界面降低使用门槛。

### EPIC-006: 性能优化
